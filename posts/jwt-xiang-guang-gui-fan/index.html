JWT相关规范

2022/01/06

简介

JWT有一组被统称为JOSE（JavaScript Object Signing & Encryption）的补充规范来处理签名和加密，由以下组件组成：

规范 作用
JWA 签名或加密算法列表
JWS 签名过程
JWE 加密过程
JWK 密钥以及密钥集的格式

我们可以认为JWT是一个接口，而JWS和JWE就是具体的实现。

JWA

JWA （JSON Web Algorithms）RFC7518 定义了一系列的算法用于后面的JWS、JWE和JWK，比如之前说过的HS256HMAC using SHA-256)、RS256(RSASSA-PKCS1-v1_5 using SHA-256)和ES256(ECDSA using P-256 and SHA-256)算法。

相关的算法可以简单参考下表：

“alg” Param Value Digital Signature or MAC Algorithm Requirements Implementation
HS256 HMAC using SHA-256 Required
HS384 HMAC using SHA-384 Optional
HS512 HMAC using SHA-512 Optional
RS256 RSASSA-PKCS1-v1_5 using SHA-256 Recommended
RS384 RSASSA-PKCS1-v1_5 using SHA-384 Optional
RS512 RSASSA-PKCS1-v1_5 using SHA-512 Optional
ES256 ECDSA using P-256 and SHA-256 Recommended+
ES384 ECDSA using P-384 and SHA-384 Optional
ES512 ECDSA using P-521 and SHA-512 Optional
PS256 RSASSA-PSS using SHA-256 and MGF1 with SHA-256 Optional
PS384 RSASSA-PSS using SHA-384 and MGF1 with SHA-384 Optional
PS512 RSASSA-PSS using SHA-512 and MGF1 with SHA-512 Optional
none No digital signature or MAC performed Optional

none表示是一个不签名的不安全的JWT。

JWS

JWS （JSON Web Signature）RFC7515 就是我们之前使用JWT实现。

JWT的第一部分header在一些文档里也被叫做JOSE header，所以在这里记住一下。

JWS 的序列化实际是有两种，JWS 紧凑序列化（JWS Compact Serialization）和 JWS JSON序列化（JWS JSON Serialization）。 只有当它遵循紧凑序列化时才称为 JWT。任何JWT都必须遵循紧凑序列化。换句话说，遵循JSON序列化的JWS不能称为JWT。

JWS紧凑序列化

Base64URLEncode(UTF8(JWS Protected Header))+"."+
Base64URLEncode(JWS Payload)+"."+
Base64URLEncode(JWS Signature)

对于紧凑序列化，Protected Header就是整个JOSE header。

JWS JSON序列化

考虑到JSON序列化并不算是JWT，这里只是简单举个例子，看看JSON序列化的格式。

     {
      "payload":"<payload contents>",
      "signatures":[
       {"protected":"<integrity-protected header 1 contents>",
        "header": "<non-integrity-protected header 1 contents>",
        "signature":"<signature 1 contents>"},
       {"protected":"<integrity-protected header N contents>",
        "header": "<non-integrity-protected header N contents>",
        "signature":"<signature N contents>"}]
     }

JWE

JWE （JSON Web Encryption）RFC7516 定义了在JWT中表示加密内容的方式。

JWE同样也是有紧凑序列化和JSON序列化两种。

JWE紧凑序列化

Base64URLEncode(UTF8(JWE Protected Header))+"."+
Base64URLEncode(JWE Encrypted Key)+"."+
Base64URLEncode(JWE Initialization Vector)+"."+
Base64URLEncode(JWE Ciphertext)+"."+
Base64URLEncode(JWE Authentication Tag)

对于紧凑序列化，Protected Header同样也是整个JOSE header。

具体的加密密钥、初始化向量(IV)、密文和认证标签生成过程就不细讲，直接略过。

JWE JSON序列化

JWE JSON序列化也仅举例说明一下。

    {
      "protected":"<integrity-protected shared header contents>",
      "unprotected":"<non-integrity-protected shared header contents>",
      "recipients":[
       {"header":"<per-recipient unprotected header 1 contents>",
        "encrypted_key":"<encrypted key 1 contents>"},
       {"header":"<per-recipient unprotected header N contents>",
        "encrypted_key":"<encrypted key N contents>"}],
      "aad":"<additional authenticated data contents>",
      "iv":"<initialization vector contents>",
      "ciphertext":"<ciphertext contents>",
      "tag":"<authentication tag contents>"
     }
